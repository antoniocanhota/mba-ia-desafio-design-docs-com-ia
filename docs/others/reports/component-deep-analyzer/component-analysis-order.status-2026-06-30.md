# Component Deep Analysis Report

**Component:** `order.status.ts`
**Path:** `src/modules/orders/order.status.ts`
**Project:** order-management-api (Order Management System REST API)
**Analysis Date:** 2026-06-30

---

## 1. Executive Summary

`order.status.ts` is a small, side-effect-free module that centralizes the **order status state machine** for the Order Management System. It has no classes, no I/O, and no framework dependencies — it is a pure-function module operating exclusively on the `OrderStatus` enum generated by Prisma (`@prisma/client`).

The module has two responsibilities, both expressed as pure predicates/lookups over a single private constant:

1. **State machine definition and queries** — a `transitions` adjacency map defines which `OrderStatus` values are reachable from each status, plus three exported helper functions (`canTransition`, `allowedTransitions`, `isTerminal`) that query this map.
2. **Stock-impact classification** — two exported predicate functions (`shouldDebitStock`, `shouldReplenishStock`) that classify a `(from, to)` transition pair as triggering inventory debit or inventory replenishment, decoupled from the transition-validity concern.

The component is consumed by exactly one caller in the codebase, `OrderService.changeStatus` (`src/modules/orders/order.service.ts:126-179`), which uses it as the single source of truth for (a) whether a requested status change is legal, and (b) whether that change should trigger a stock mutation before persisting the new status and writing history.

Key findings:
- The module is fully deterministic, has no external dependencies beyond the Prisma-generated `OrderStatus` type, and is trivially testable in isolation — yet **no dedicated unit test file exists** for it. All coverage is indirect, via HTTP-level integration tests in `tests/orders.test.ts` that exercise `OrderService.changeStatus`.
- The state machine graph, expressed as data (`transitions` record), is a linear/mostly-linear fulfillment pipeline (`PENDING → PAID → PROCESSING → SHIPPED → DELIVERED`) with `CANCELLED` reachable as a side branch from `PENDING`, `PAID`, and `PROCESSING` only — not from `SHIPPED` or `DELIVERED`.
- `DELIVERED` and `CANCELLED` are both terminal states (`isTerminal` returns `true` for both, since both map to empty transition arrays).
- Same-status "transitions" (`from === to`) are not explicitly rejected by `canTransition` itself — that guard lives in the caller (`order.service.ts:140-146`), a coupling point worth noting (see Technical Debt section).
- The stock-impact rules (`shouldDebitStock`, `shouldReplenishStock`) are independent of the transition-validity graph — they do not call `canTransition` and would return `true` for a stock-affecting pair even if that pair were not a legal transition. Correctness therefore depends on the caller always checking `canTransition` first, which it does.

## 2. Data Flow Analysis

The component itself has no "entry/exit" flow of its own (it is a pure utility), but it sits inside the following request-level data flow when a client changes an order's status:

```
1. HTTP PATCH /api/v1/orders/:id/status enters via order.routes.ts
2. Request validated (params + body) by validate() middleware using order.schemas.ts (updateOrderStatusSchema: toStatus must be a valid OrderStatus enum value)
3. OrderController.changeStatus (order.controller.ts:38-46) delegates to OrderService.changeStatus
4. OrderService.changeStatus (order.service.ts:126-179) opens a Prisma transaction:
   a. Loads the order + items from the database (from = order.status)
   b. Same-status guard: if (from === to) throws ConflictError('INVALID_STATUS_TRANSITION') -- NOTE: this check is performed by the caller, not by order.status.ts
   c. canTransition(from, to) [order.status.ts:12-14] is called -- looks up transitions[from].includes(to)
      - If false: caller throws InvalidStatusTransitionError(from, to) -> HTTP 409
   d. shouldDebitStock(from, to) [order.status.ts:29-31] is called
      - If true (PENDING -> PAID only): caller invokes debitStock(), which re-reads product stock and throws InsufficientStockError (HTTP 422) if any item's stockQuantity < requested quantity; otherwise decrements stock
   e. shouldReplenishStock(from, to) [order.status.ts:33-37] is called
      - If true (CANCELLED from PAID or PROCESSING): caller invokes replenishStock(), incrementing product stock for each order item
   f. Order.status is updated to `to` in the database
   g. OrderStatusHistory row is inserted (fromStatus, toStatus, changedById, reason)
5. Refreshed order (with items/history/customer) is returned to OrderController
6. Response formatted as JSON with 200 status
```

The module's pure functions are called synchronously, in-memory, with no network or database access of their own — all side effects (stock mutation, persistence) happen in the caller (`order.service.ts`), guarded by the boolean results these functions return.

### Business Rule: Order Status Transition Graph

**Overview:**
The `transitions` constant (order.status.ts:3-10) is a `Readonly<Record<OrderStatus, ReadonlyArray<OrderStatus>>>` that defines, for every possible order status, the complete and exhaustive set of statuses it may legally transition to. This is the single source of truth for order lifecycle validity in the entire system.

**Detailed description:**
The order lifecycle is modeled as a directed graph over the six `OrderStatus` enum values (`PENDING`, `PAID`, `PROCESSING`, `SHIPPED`, `DELIVERED`, `CANCELLED`). The primary path is linear and represents the normal fulfillment pipeline: an order is created in `PENDING`, becomes `PAID` once payment is confirmed, moves to `PROCESSING` while being prepared for shipment, becomes `SHIPPED` once dispatched, and finally reaches `DELIVERED` once the customer receives it. Each of the first three states (`PENDING`, `PAID`, `PROCESSING`) also has a secondary edge to `CANCELLED`, representing the ability to cancel an order at any point before it physically leaves the warehouse.

Critically, `CANCELLED` is **not reachable** from `SHIPPED`. Once an order has been dispatched to a carrier, the module's graph treats cancellation as no longer a valid business operation — the only legal next state from `SHIPPED` is `DELIVERED`. This is a deliberate business constraint (not an oversight, since `SHIPPED` explicitly appears in the graph with only one edge) and implies that any real-world "shipped order that must be cancelled" scenario (e.g., a return-to-sender or lost-package process) is out of scope for this state machine and would need to be modeled as a different flow entirely (not evidenced in this codebase).

`DELIVERED` and `CANCELLED` both map to empty arrays, making them terminal nodes with no outgoing edges — this is what powers the `isTerminal` helper. The graph is asymmetric and one-directional: there is no path to move an order backward (e.g., `PAID` back to `PENDING`, or `SHIPPED` back to `PROCESSING`). Every transition not explicitly listed as an outgoing edge — including all "same status" pairs, e.g. `PENDING -> PENDING` — is implicitly invalid according to this graph, since `canTransition` merely checks array membership.

**Rule workflow:**
```
canTransition(from, to):
  1. Look up transitions[from] (guaranteed to exist -- the Record type covers all 6 enum values exhaustively)
  2. Return true if `to` is present in that array, false otherwise
  3. Caller (OrderService.changeStatus) throws InvalidStatusTransitionError (HTTP 409, code INVALID_STATUS_TRANSITION) when false
```

Valid transitions (7 total edges in the graph):
- `PENDING -> PAID`
- `PENDING -> CANCELLED`
- `PAID -> PROCESSING`
- `PAID -> CANCELLED`
- `PROCESSING -> SHIPPED`
- `PROCESSING -> CANCELLED`
- `SHIPPED -> DELIVERED`

All other 29 of the 36 possible `(from, to)` pairs (including same-state pairs) are invalid.

### Business Rule: PENDING -> PAID Stock Debit

**Overview:**
`shouldDebitStock` (order.status.ts:29-31) determines whether a given transition should trigger inventory decrement. It is hardcoded to a single transition pair, defined by the `STOCK_DEBIT_TRANSITION` constant (order.status.ts:24-27): `from = PENDING`, `to = PAID`.

**Detailed description:**
The business model treats payment confirmation as the moment inventory is committed to a customer. While an order sits in `PENDING`, no stock is reserved or reduced — this means multiple pending orders could theoretically over-commit the same inventory (this module does not implement reservation/holding logic; it only reacts at the moment of the `PENDING -> PAID` transition). Only when the order actually transitions from `PENDING` to `PAID` does the system consider the stock "spent," at which point `OrderService.debitStock` (order.service.ts:204-231) is invoked by the caller to decrement `Product.stockQuantity` for every line item.

This rule is intentionally narrow: it fires only for the exact `PENDING -> PAID` pair and no other pair, even though other transitions exist in the graph. For example, there is no `shouldDebitStock` logic for `PAID -> PROCESSING` or `PROCESSING -> SHIPPED`, meaning inventory is debited exactly once, at the earliest possible point after payment confirmation, and never again later in the fulfillment pipeline. The `STOCK_DEBIT_TRANSITION` object is exported as a named constant, suggesting an intent to allow other parts of the system (or future code) to introspect which specific transition triggers debit, rather than hardcoding the pair inline in multiple places.

Because `shouldDebitStock` does not itself verify that `(from, to)` is a *valid* transition per the graph, this function is only safe to call after `canTransition` has already confirmed legality — which is exactly the order of operations followed in `OrderService.changeStatus` (order.service.ts:147, then 151). If this ordering were ever reversed or the `canTransition` check removed, `shouldDebitStock` would still (incorrectly) report `true` for a `PENDING -> PAID` pair regardless of whether that specific transition was reachable, since the two functions are structurally decoupled with no shared invariant enforcement.

**Rule workflow:**
```
shouldDebitStock(from, to):
  1. Compare from === STOCK_DEBIT_TRANSITION.from (PENDING) AND to === STOCK_DEBIT_TRANSITION.to (PAID)
  2. Return true only for that exact pair
  3. Caller invokes debitStock(tx, order.items):
     a. Re-fetches current Product rows for all items in the same transaction
     b. For each item, checks product.stockQuantity >= item.quantity
     c. If any item has insufficient stock, collects {sku, requested, available} and throws InsufficientStockError (HTTP 422, code INSUFFICIENT_STOCK) -- entire transaction rolls back, no partial debit occurs
     d. If all items have sufficient stock, decrements stockQuantity for every item within the same DB transaction as the status update and history insert
```

### Business Rule: Cancellation Stock Replenishment

**Overview:**
`shouldReplenishStock` (order.status.ts:33-37) determines whether a transition should trigger inventory increment (giving stock back). It returns `true` when the destination status is `CANCELLED` and the origin status is either `PAID` or `PROCESSING`.

**Detailed description:**
This rule exists to reverse the effect of the stock-debit rule described above. Since stock is only debited at `PENDING -> PAID`, any subsequent cancellation from a state where the order has already had stock debited (`PAID` or `PROCESSING`) must give that stock back to avoid inventory shrinkage with no corresponding sale. The rule explicitly excludes `PENDING -> CANCELLED`: cancelling an order that never left `PENDING` must **not** replenish stock, because no debit ever occurred for it in the first place. This asymmetry is not accidental — it directly mirrors the debit rule's boundary and is confirmed by the integration test `replenishes stock when going PAID -> CANCELLED` (tests/orders.test.ts:134-161), which explicitly asserts stock is restored to its pre-debit level after a `PAID -> CANCELLED` transition.

Because `SHIPPED -> CANCELLED` is not a valid edge in the transition graph at all (see the Transition Graph rule above), the replenishment rule never needs to consider a `from: SHIPPED` case — an order that has shipped cannot be cancelled through this state machine, so there is no scenario in which shipped-and-already-picked stock would need replenishing via this function. This is a subtle but important interaction between the two rule sets: the stock-impact functions are only ever invoked (by the current caller) after `canTransition` has confirmed the pair is legal, so `shouldReplenishStock`'s logical domain is implicitly narrowed to just the two valid cancellation edges (`PAID -> CANCELLED`, `PROCESSING -> CANCELLED`) even though the function itself does not perform that narrowing.

The replenishment operation itself (`OrderService.replenishStock`, order.service.ts:233-243) is unconditional once triggered — unlike the debit path, it does not check for a maximum stock ceiling or any other constraint; it simply increments `stockQuantity` for every order item by the ordered quantity, within the same transaction as the status update.

**Rule workflow:**
```
shouldReplenishStock(from, to):
  1. Check to === CANCELLED
  2. Check from === PAID OR from === PROCESSING
  3. Return true only if both conditions hold (i.e., true for {PAID->CANCELLED, PROCESSING->CANCELLED}; false for PENDING->CANCELLED and all other pairs)
  4. Caller invokes replenishStock(tx, order.items):
     a. For each item, increments Product.stockQuantity by item.quantity
     b. No stock-ceiling or availability check is performed on replenishment
```

### Business Rule: Terminal State Detection

**Overview:**
`isTerminal` (order.status.ts:20-22) reports whether a given status has no further legal transitions, derived directly from the same `transitions` map used for validity checking (not a separately maintained list).

**Detailed description:**
A status is considered terminal if and only if its entry in the `transitions` record is an empty array. Under the current graph, exactly two statuses are terminal: `DELIVERED` and `CANCELLED`. This function is exported but, per the dependency analysis below, **has no callers anywhere in the current codebase** — it is dead code from a usage perspective, though it remains part of the module's public contract and is exercised only implicitly (its correctness is a direct structural consequence of the `transitions` map, so any test of the map's terminal entries would indirectly validate it, but no such test exists).

Because terminality is derived rather than duplicated, this design guarantees `isTerminal` can never drift out of sync with the transition graph itself — adding or removing outgoing edges for `DELIVERED` or `CANCELLED` automatically changes `isTerminal`'s behavior for free, with no separate maintenance burden. This is a positive structural property of the implementation, even though the function is currently unused.

**Rule workflow:**
```
isTerminal(status):
  1. Look up transitions[status]
  2. Return true if the resulting array's length is 0, false otherwise
  3. (No current caller integrates this into request-handling flow)
```

### Business Rule: Same-Status Transition Rejection (caller-side, related rule)

**Overview:**
Not implemented inside `order.status.ts` itself, but directly adjacent to it: `OrderService.changeStatus` (order.service.ts:140-146) explicitly rejects a request where `toStatus === order.status` (i.e., `from === to`) with a `ConflictError` before ever calling `canTransition`.

**Detailed description:**
This rule is documented here because it is functionally part of the "transition validity" concern that `order.status.ts` otherwise fully owns, yet it is NOT implemented within the analyzed component. Since no status in the `transitions` map lists itself as a valid destination (e.g., `PENDING` does not include `PENDING` in its array), `canTransition(from, from)` would already correctly return `false` for every status without this extra guard — making the caller's explicit `from === to` check redundant with (but not contradicted by) the pure function's own logic. The caller nonetheless implements it separately, using the same `INVALID_STATUS_TRANSITION` error code as a true `canTransition` failure, so the two code paths are behaviorally indistinguishable from the API consumer's perspective (both yield HTTP 409 with the same error code) even though only one of them actually executes `canTransition`.

This is flagged as a coupling/redundancy observation rather than a functional bug: the business rule ("you cannot transition to the same status you're already in") is enforced correctly today, but it is enforced by duplicated logic split across two files, with the pure state machine module not being the sole authority for this specific sub-case.

**Rule workflow:**
```
OrderService.changeStatus (caller):
  1. from = order.status (current DB value); to = input.toStatus (request body)
  2. if from === to: throw ConflictError('INVALID_STATUS_TRANSITION') -- short-circuits before canTransition is ever called
  3. else: proceed to canTransition(from, to) as normal
```

## 8. Design Patterns & Architecture

| Pattern | Implementation | Location | Purpose |
|---------|----------------|----------|---------|
| State Machine (table-driven / adjacency-list) | `transitions` Record mapping each status to its allowed destinations | order.status.ts:3-10 | Encodes the entire order lifecycle as declarative data rather than imperative if/else or switch logic |
| Pure Function / Functional Core | All exported functions (`canTransition`, `allowedTransitions`, `isTerminal`, `shouldDebitStock`, `shouldReplenishStock`) are deterministic, side-effect-free, and depend only on their arguments | order.status.ts:12-37 | Isolates business rules from I/O ("functional core, imperative shell" — the imperative shell being `order.service.ts`) |
| Named Constant for Magic Pair | `STOCK_DEBIT_TRANSITION` object instead of inline `from === 'PENDING' && to === 'PAID'` | order.status.ts:24-27 | Gives the specific debit-triggering pair a documented, reusable, introspectable identity |
| Predicate/Guard Functions | Boolean-returning functions used by the caller purely as gates before performing an action | order.status.ts:12,29,33 | Separates "should I do X" (decision) from "do X" (action, implemented in order.service.ts) |
| Readonly / Immutable Data Modeling | `Readonly<Record<...>>`, `ReadonlyArray<...>`, `as const` | order.status.ts:3, 16, 27 | Prevents accidental mutation of the state machine definition at the type level |

Architectural note: the module deliberately separates two concerns that could have been merged (transition validity vs. stock impact) into two independently callable rule sets. This grants the caller flexibility (e.g., it could check `canTransition` without touching stock, or vice versa) but also means the two rule sets have no enforced mutual consistency at the type or runtime level — see Technical Debt below.
